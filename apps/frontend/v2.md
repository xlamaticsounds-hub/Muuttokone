# v2: Self‑hosted migration – stupid simple battle plan

Goal: Move to self‑hosted PostgreSQL + Directus (admin/CRM), keep Sanity for blog, drop external managed services. Keep it incremental and low‑risk.

## Decisions (for now)
- DB: PostgreSQL (self‑hosted)
- Admin/CRM: Directus at mgmt.muuttokone.fi (or crm.muuttokone.fi if you prefer the URL)
- Blog: Keep Sanity as‑is for now
- Caching/queues: none (can add later)
- File storage:
  - Prefer Directus “local filesystem” storage for admin-managed files (simple, self‑hosted)
  - If you want DB‑only storage for quote attachments, use a custom table with `bytea` in our API (Directus won’t natively store file binaries in PostgreSQL; it stores metadata in DB and bytes on disk). We can mix: Directus for CRM + disk, custom API for DB‑stored attachments if required.

## Phases (checkbox = completion target)
1) Infra baseline
- [ ] Home server: Docker + compose, reverse proxy (Traefik/Caddy), TLS
- [ ] PostgreSQL container + volume + backups
- [ ] (Optional) Cloudflare DNS/Tunnel for public hostnames

2) Data layer & app connectivity
- [ ] Create Postgres DB/users: `app` (Next/Prisma), `directus`
- [ ] Point your app’s `DATABASE_URL` to self‑hosted Postgres
- [ ] Run Prisma generate/migrate in app (no schema changes yet)

3) Directus (admin/CRM)
- [ ] Bring up Directus at mgmt.muuttokone.fi, connect to Postgres
- [ ] Configure local storage (disk) for files
- [ ] Create roles: Admin (full), Manager (edit CRM), Reader (read configs)
- [ ] Collections (minimal viable CRM):
  - `configs` (key: string, value: JSON) — for site config like phone/email/opening hours
  - `leads` (name, phone [required], email, is_business, business_id, notes, created_at)
  - `quotes` (lead -> relation, from_zip, to_zip, size, date, date_flex, elevator, distance, services [csv/json], inventory, address_from_extra, address_to_extra, contact_notes, status)
  - `files` (Directus built-in) + `quote_files` (m2m: quote <-> files)
  - `activities` (lead -> relation, type, content, at, by)

4) Frontend/API integration (incremental)
- [ ] Keep Sanity for blog unchanged
- [ ] Quote funnel: point POST to your existing Next.js `/api/quote`
- [ ] Inside `/api/quote`, forward payload+images into Directus:
  - Create/lookup lead, create quote, upload files to Directus Files, link via `quote_files`
  - Or (if you insist on DB‑only storage) store images in your own `attachments` table with `bytea`, and link record IDs in Directus (note: admin won’t preview DB‑stored binaries)
- [ ] Site config: FE server‑fetch from Directus `/items/configs?filter[key]=contact|brand…` and cache in memory per request (no external cache). Keep `src/config/site.ts` as fallback until switched.

5) Admin UX
- [ ] Use Directus Collections/Interfaces to manage configs/leads/quotes
- [ ] Add simple “Quote board” (status: new/in review/sent/won/lost) using Directus display presets

6) Observability & backups
- [ ] Nightly `pg_dump` (retain 7–14 days) + config export from Directus
- [ ] Log retention via the reverse proxy; add Grafana/Prometheus later if desired

7) Cutover + rollback
- [ ] Feature flag in app: toggle Directus config on/off
- [ ] When Directus is ready: switch FE to fetch configs from Directus
- [ ] If issues: toggle back to local config and continue iterating

## Minimal contracts (so FE doesn’t stall)
- Quote submit (current): POST `/api/quote` (multipart: `payload` JSON + `images[]`)
  - Server: creates/updates lead + quote in Directus; uploads files; returns `{ ok: true, id: <quote_id> }`
- Config fetch (server-side): GET Directus `/items/configs?filter[key][_in]=brand,contact` → map to the shape in `site.ts`

## Env to prep
- App: `DATABASE_URL`, `DIRECTUS_URL`, `DIRECTUS_STATIC_TOKEN` (service token for API), Sanity vars stay
- Directus: DB creds, ADMIN email/pass, storage path, public URL (for assets)

## If Directus isn’t enough
- Plan B: Stand up `crm.muuttokone.fi` using NextAdmin Pro template
  - Keep the same Postgres; Prisma for schema; reuse app service token for FE integration
  - You can still keep Directus only for configs if you like its UI better for that

## Rough day-by-day (suggested)
- Day 1: Infra + Postgres + Directus up; create collections & roles
- Day 2: Wire `/api/quote` to Directus; store attachments; basic permission rules
- Day 3: Configs in Directus + FE read path behind flag; smoke tests; backups
- Day 4+: Replace local `site.ts` with Directus values (or keep as fallback), refine CRM UIs
